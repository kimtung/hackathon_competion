Stock Exchange — QA Bug Report

Vai trò: QA engineer — mục tiêu phá hệ thống và phát hiện bug ẩn.
Phạm vi: exchange/engine (Python + uv) & client (React + Vite).
Focus: edge cases · sequence-based bugs · concurrency · rare conditions.
Báo cáo này viết trên trạng thái sau khi đã áp dụng các fix của bug_dev.html.

Phương pháp test

  • Edge cases: qty cực lớn (10^18), price=0, cl_ord_id rỗng/trùng, field null, ký tự lạ trong symbol.
  • Sequence: place → partial fill → re-submit same order; place → update config → existing resting còn hay hết.
  • Concurrency: gửi song song từ 2 WS client cùng một thời điểm, cùng account match với chính mình.
  • Rare conditions: admin đổi floor/ceiling/step trong khi có lệnh đang resting; spam 1000+ lệnh liên tục.

BUG-QA-01: Self-trade (wash trade) không bị chặn medium report only

exchange/engine/src/engine/order_book.py — _match()

Mô tả

Engine không kiểm tra buy_order.account == sell_order.account trước khi khớp. Một tài khoản có thể tự mua tự bán, dẫn tới (1) phát sinh volume giả (wash trade), (2) tác động giá nhân tạo, (3) rủi ro tuân thủ. Nhiều sàn thật triển khai STP (self-trade prevention): khi lệnh mới cross với lệnh của chính account đó, hủy một trong hai.

Điều kiện kích hoạt: Cùng một account đặt cả BUY và SELL cross giá nhau (bất kể cl_ord_id khác).

Kịch bản tái hiện

  1. Mở market.
  2. Submit SELL LIMIT 55000 qty=100, account="SAME".
  3. Submit BUY LIMIT 55000 qty=100, account="SAME".
  4. Thực tế: sinh 1 trade giữa SAME↔SAME, tăng volume ảo. Kỳ vọng: một trong hai bị hủy (STP).

Vì sao để dạng report-only

Toàn bộ test suite hiện tại dùng cùng account="ACC1" cho cả hai phía (test_exact_match, test_fill_reports, v.v.). Bật STP mặc định sẽ vi phạm kỳ vọng test. Đề xuất: thêm flag ExchangeConfig.prevent_self_trade (default False) để bật tùy ngữ cảnh.

Đề xuất automation

async def test_stp_enabled():
    engine.config.prevent_self_trade = True
    engine.submit_order(make(account="A", side=SELL, ...))
    r = engine.submit_order(make(account="A", side=BUY,  ...))
    assert len(r.trades) == 0
    assert any(er.exec_type == CANCELLED for er in r.exec_reports)

BUG-QA-02: Duplicate cl_ord_id được accept silent medium fixed

exchange/engine/src/engine/matching.py — submit_order()

Mô tả

FIX spec yêu cầu ClOrdID (11) là unique trên một session. Engine hiện tại không tracking — gửi 2 order có cùng cl_ord_id đều được accept. Hệ quả: (1) audit trail bị nhầm lẫn, (2) sau fix BUG-04, mapping _order_owners[cl_ord_id] bị ghi đè — exec report của lệnh thứ nhất có thể bị gửi nhầm cho client thứ hai hoặc ngược lại.

Điều kiện kích hoạt: 2 message new_order với cùng cl_ord_id được submit, có thể cùng hoặc khác ws client.

Kịch bản tái hiện

  1. Client A submit {"cl_ord_id":"DUP1", "symbol":"FPT", "side":"BUY", "price":55000, "quantity":100}.
  2. Client B submit {"cl_ord_id":"DUP1", "symbol":"FPT", "side":"SELL", "price":60000, "quantity":200}.
  3. Thực tế: cả hai NEW. Lệnh của A có thể khớp sau đó nhưng exec report lại được route tới B (ghi đè mapping).

Fix áp dụng

Giữ set _seen_cl_ord_ids (hoặc mapping) trong MatchingEngine. Nếu trùng → reject với reason Duplicate cl_ord_id trước khi route tới book.

BUG-QA-03: Field null hoặc không phải số gây TypeError, drop connection high fixed

exchange/engine/src/engine/ws_server.py — _handle_new_order()

Mô tả

Handler bọc try/except (KeyError, ValueError) nhưng KHÔNG bắt TypeError. Khi client gửi "quantity": null hoặc "price": null, int(None) raise TypeError, bubble lên _handle_client's async-for → kết thúc loop → connection đóng đột ngột (không kèm error frame cho client).

Tương tự với "symbol": 123 (number thay vì string) — .upper() raise AttributeError, cũng không được bắt.

Điều kiện kích hoạt: Client gửi JSON với field ép kiểu bị fail (không phải string/number), hoặc null tại quantity/price/symbol.

Kịch bản tái hiện

  1. Mở WS, gửi {"type":"new_order","cl_ord_id":"X","symbol":"FPT","side":"BUY","ord_type":"LIMIT","price":55000,"quantity":null}.
  2. Kết quả: connection đóng không có thông báo, client phải reconnect.

Fix áp dụng

Mở rộng except thành (KeyError, ValueError, TypeError, AttributeError) và trả về error frame.

BUG-QA-04: Không có rate-limit / max-orders trên WS medium report only

exchange/engine/src/engine/ws_server.py

Mô tả

Một client duy nhất có thể gửi hàng ngàn order/giây. Mỗi order resting thêm vào deque trong book — bộ nhớ tăng tuyến tính, _trades cũng unbounded. Client độc hại có thể DoS bằng cách spam lệnh hợp lệ rồi không bao giờ cancel.

Điều kiện kích hoạt: Attacker loop gửi new_order liên tục không delay.

Kịch bản tái hiện

  1. Script Python gửi 100k order LIMIT giá rải đều trong floor–ceiling, qty hợp lệ.
  2. Theo dõi RAM của engine process — tăng đều; latency cho client khác cũng tăng.

Đề xuất

  • Token bucket: mỗi client ≤ N message/s.
  • Cap max_open_orders_per_account.
  • Eviction policy cho _trades cũ (theo thời gian hoặc count).

BUG-QA-05: Đổi config không hủy các lệnh resting đã vi phạm medium fixed

exchange/engine/src/engine/matching.py — update_stock_config()

Mô tả

Admin hạ ceiling / nâng floor / đổi price_step/qty_step khi có lệnh đang resting. Lệnh cũ vi phạm config mới vẫn nằm trên sổ và tiếp tục match — trạng thái sổ và config trở nên không nhất quán.

Điều kiện kích hoạt: Có lệnh resting tại giá nằm ngoài [new_floor, new_ceiling] hoặc không align step mới, và admin gọi PUT /api/stocks/{symbol}.

Kịch bản tái hiện

  1. Floor ACB=20000, ceiling=30000. Submit BUY LIMIT 22000 qty=100.
  2. Admin gọi PUT /api/stocks/ACB {"floor": 25000}.
  3. Thực tế: bid 22000 vẫn tồn tại, có thể khớp với SELL tới sau (SELL ở giá 22000 sẽ bị reject do dưới floor mới, nhưng SELL ở 25000 có thể vẫn cross với bid 22000 qua giá khớp best_price 22000 — dưới floor, vi phạm).
  4. Kỳ vọng: lệnh 22000 phải bị hủy CANCELLED khi config đổi.

Fix áp dụng

Thêm OrderBook.purge_invalid_orders() trả về danh sách đã hủy; gọi trong MatchingEngine.update_stock_config(). Các lệnh invalid được cancel(), log để observability.

BUG-QA-06: Quantity/price cực lớn bypass validation medium fixed

exchange/engine/src/engine/config.py — validate_quantity()

Mô tả

validate_quantity chỉ check > 0 và align step. Không có max cap. Có thể submit qty=10^18. Python xử lý int tùy ý nhưng:

  • Frontend JS dùng Number (double 64-bit) → mất chính xác ở số > 2^53 ≈ 9e15.
  • JSON serialize 10^18 → số chính xác tại Python, nhưng parse trên JS sẽ rounding.
  • UI hiển thị sai, có thể tạo fill giả (nếu admin đổi ceiling rất lớn kết hợp).
Điều kiện kích hoạt: Client gửi quantity hoặc price lớn hơn max an toàn.

Kịch bản tái hiện

  1. Submit BUY LIMIT qty=10**18 ở 55000 FPT — accept OK.
  2. Client JS nhận message, JSON.parse ra 1e+18, toLocaleString() mất chính xác.

Fix áp dụng

Thêm MAX_QUANTITY = 1_000_000_000 trong StockConfig.validate_quantity() và reject nếu vượt.

BUG-QA-07: Auto-gen vẫn gửi order khi market CLOSED low fixed

client/src/components/OrderEntry.jsx — generateRandomOrder

Mô tả

Khi người dùng bật "Start Auto" nhưng market đang đóng, client vẫn bắn order — server reject hàng loạt, spam cả comm_logs lẫn UI execution reports. Không có cảnh báo sớm cho user.

Điều kiện kích hoạt: Admin chưa open market (hoặc đã close), user bật Auto Order.

Fix áp dụng

Trong generateRandomOrder, kiểm tra snap.market_state === "OPEN" trước khi gửi, nếu không thì skip (vẫn tick interval nhưng no-op).

BUG-QA-08: execReports không reset khi reconnect low fixed

client/src/hooks/useWebSocket.js — onclose

Mô tả

Fix BUG-09 của dev report đã reset orderBooks & trades khi onclose, nhưng execReports thì không. User reconnect xong vẫn thấy exec reports cũ lẫn vào reports mới, khó phân biệt.

Điều kiện kích hoạt: Client mất connection rồi reconnect trong khi state React vẫn còn (không refresh trang).

Fix áp dụng

Thêm setExecReports([]) trong onclose.

BUG-QA-09: Không có API cancel/modify — sequence place→partial→cancel→modify không thể tái hiện medium report only

exchange/engine (feature gap)

Mô tả

Engine hỗ trợ new_order nhưng không có cancel_order (FIX F) hoặc order_cancel_replace (FIX G). User bị partial fill không thể cancel phần resting, cũng không thể modify giá/quantity. Trong một sàn thực tế đây là tính năng bắt buộc.

Điều kiện kích hoạt: Bất kỳ lúc nào user muốn đổi ý về một lệnh đã resting.

Kịch bản mong đợi (hiện tại impossible)

  1. Place BUY LIMIT 55000 qty=500.
  2. Partial fill 200 → còn 300 resting.
  3. Gửi cancel → không có endpoint. User phải tự đặt lệnh đối ứng để neutralize.
  4. Modify (ví dụ hạ price xuống 54000) → không có endpoint.

Đề xuất automation

async def test_cancel_partial_fill():
    # place BUY 500 @55000
    # send matching SELL 200 @55000 → partial
    await ws.send({"type":"cancel_order","cl_ord_id":"B1"})
    er = await recv_json(ws)
    assert er["exec_type"] == "CANCELLED"
    assert er["leaves_qty"] == 0
    assert er["cum_qty"] == 200  # preserved

BUG-QA-10: onerror nuốt lỗi không log low fixed

client/src/hooks/useWebSocket.js — ws.onerror = () => {}

Mô tả

Callback rỗng khiến lỗi WS (DNS fail, TLS handshake fail, ICE disconnect) bị nuốt sạch. QA khó debug khi reconnect liên tục thất bại.

Fix áp dụng

Log console.warn("WebSocket error", e) để hỗ trợ điều tra.

BUG-QA-11: Re-submit cùng Order instance sau khi FILLED — idempotent kém low report only

exchange/engine/src/engine/order_book.py — process_order()

Mô tả

Nếu pháp trình sử dụng engine truyền cùng một Order object hai lần vào submit_order: lần 2 gán order_id mới (tiêu tốn counter), validate lại, match-loop no-op do leaves_qty=0. Không có lỗi hay báo, nhưng counter bị tiêu — tạo "ORD-X-N" skip số.

Đề xuất

Khi order.status đã ở trạng thái terminal (FILLED/CANCELLED/REJECTED) thì reject ngay với lý do Order already finalized. Ít thật sự xảy ra trong production (vì order luôn được tạo mới từ message), nhưng giúp phòng thủ ở API nội bộ.

Đề xuất automation tổng thể

  • Viết một bộ tests/test_qa_edge.py gom các case trên (@pytest.mark.parametrize qty=[0, -1, 10^18, 10^9+1], v.v.).
  • Thêm test async concurrency: dùng asyncio.gather để submit N order song song cùng cl_ord_id.
  • Thêm fuzz test JSON bằng hypothesis: sinh ngẫu nhiên payload để phát hiện những TypeError/AttributeError chưa cover.
  • E2E: Playwright mô phỏng 2 tab + admin close market giữa lúc auto-gen đang bật → verify UI không freeze, không spam reject vô hạn.
